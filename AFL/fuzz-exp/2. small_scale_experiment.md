## Small-scale experiment 

### 1. 진행 과정 
(1) 도커 가상 실험 환경 진입   
```
cd ~/afl/fuzz-exp
sudo ./launch.sh
```   
(2) 5분 타임아웃 퍼징 가동  
```
./run_fuzzer.sh afl 5m libpng_read_fuzzer png "@@" ""
```
- `5m` : 무한히 동작하는 퍼저를 5분 뒤 강제 종료시켜 소규모 테스트로 변환
- `libpng_read_fuzzer` & `png` : 공격할 타겟 프로그램과 정상적인 초기 입력값(Seed)
- "@@": AFL 퍼저가 원본 png 파일을 이리저리 변형시킨 뒤, 그 조작된 파일들을 타겟 프로그램에 밀어 넣을 위치를 지정해 주는 특수 기호
   
(3) 실행 로그  
- 명령어 실행 직후 화면이 멈춘 것처럼 보임
- 5분 뒤에 변이된 파일들을 재실행하는 과정이 출력됨

(4) 결과 해석   
- 실험 종료 후 `/output` 폴더에 생성된 결과로 퍼징 성과 확인
- 취약점 탐지 결과
  - `crashes/` : 퍼저가 타겟 프로그램에 치명적인 오류를 유발해 강제 종료시킨 입력 파일들이 모인 곳
  - `hangs/` : 프로그램을 무한 루프에 빠뜨리거나 응답 불능 상태로 만든 입력 파일들 (DoS와 연관될 수 있음)
- 학습 데이터 및 변이 결과
  - `queue/` : 프로그램의 새로운 경로를 발견하는 데 성공한 파일들을 저장 (파일명에 어떤 변이 기법을 썼는지 기록되어있음)
  - `tc_birth.csv` : `queue`에 저장된 돌연변이 테스트케이스들이 언제, 어떤 방식으로 생성되었는지를 시간순으로 기록한 파일
- 통계 및 커버리지 지표
  - `fuzzer_stats` : 초당 몇 번의 실행을 했는지(execs_per_sec), 총 몇 번 실행했는지, 총 몇 개의 경로를 찾았는지 등 전반적인 성능 지표가 텍스트로 요약되어 있음
  - `coverage.txt` : **타겟 프로그램 전체 소스코드 중 몇 퍼센트를 실행해 보았는지(Code Coverage)** 를 수치로 기록한 커버리지 측정의 최종 결과물
  

### 2. 쉘 스크립트 구조
  - `run_fuzzer.sh`
    - `$1 <fuzzer>` : `afl` (사용할 퍼저 지정)
    - `$2 <timelimit>` : `5m` (소규모 테스트를 위한 타임아웃 시간)
    - `$3 <prog>` : `libpng_read_fuzzer` (타겟 프로그램 이름)
    - `$4 <seed>` : `png` (초기 입력값 폴더)
    - `$5 <cmdline>` : `"@@"` (퍼저가 입력값을 파일로 치환할 위치 지정)
    - `$6 <input_file>` : `""` (추가 파일 입력 여부)

- AFL 퍼저를 통해 생성된 `crashes/` 폴더 내의 테스트 케이스들을 ASAN이 적용된 타겟 바이너리에 자동으로 리플레이하고, ASAN 에러 로그를 추출하여 호스트 시스템에 저장하는 파이썬 자동화 스크립트(`scripts/replay_crash.py`) 작성

### 구현 과정 
STEP 1) 도커 컨테이너를 띄우고 실험 결과(`crashes/`)를 컨테이너 내부로 복사  
STEP 2) 크래시 파일들을 하나씩 타겟 바이너리에 입력하여 실행하고 발생하는 ASAN 로그를 `crash.log` 파일에 저장   
STEP 3) 누적된 `crash.log` 파일을 호스트의 실험 디렉토리로 추출 

### Trial & Error
(1) Permission Denied   
- `subprocess.run()`으로 docker cp 명령어를 실행할 때 권한 거부 에러 발생
- 원인 : 리눅스 환경에서 도커 데몬에 접근하기 위한 `sudo` 권한이 누락됨
- 해결 : 스크립트 내의 모든 `docker` 명령어 앞에 `sudo`를 명시적으로 추가

(2) 컨테이너 이름 충돌 
- 스크립트 재실행 시 `The container name "/replay-objdump-0" is already in use` 에러 발생
- 원인 : 이전 실행에서 정상 종료되지 않았거나 남겨진 동일한 이름의 컨테이너가 존재함
- 해결 : 스크립트 실행 전, 기존의 `replay-` 접두사가 붙은 컨테이너를 정리하는 명령어 실행
  ```
  sudo docker rm -f $(sudo docker ps -a -q --filter "name=replay-")
  ```

(3) 타겟 바이너리 경로 탐색
- 도커 내부에서 리플레이 실행 시 `sh: 1: /idlc: not found` 에러 발생
- 원인 : 실행 바이너리의 경로를 제대로 파악하지 못함
- 해결 : `find / -type f -name "idlc"` 명령어로 컨테이너 내부를 수색하여 실제 경로를 찾아내 변수에 반영

(4) 인자 전달 방식의 차이 
- 타겟 프로그램이 크래시를 일으키지 않고 `Usage: idlc [OPTIONS] FILE` 도움말만 띄우고 종료됨
- 원인 : `cat [파일] | [프로그램]` 형태의 파이프라인(Stdin)으로 데이터를 밀어넣었으나, 타겟 프로그램은 인자로 파일 경로를 직접 넘겨받는 방식 (File Argument)을 요구함
- 해결 : 명령어를 `/[프로그램] [파일경로]` 형태로 수정
- <mark>*</mark> : 프로그램마다 입력값(인자)을 어떻게 넣어야 하는지 파악하는 것은 퍼징 환경 구축의 핵심이므로 인자 전달 방식의 차이를 확실하게 기억해야 함

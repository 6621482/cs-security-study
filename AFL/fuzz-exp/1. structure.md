## Structure

### 1. 핵심 디렉터리 및 파일 구조 
- `docker-setup/` : 가상 실험 환경을 구축하기 위한 핵심 스크립트와 도구들이 모여있는 디렉터리
  - `AFL @ 8f5468e` : 퍼징을 수행하는 메인 퍼저 (Git Submodule로 연결되어 있어 특정 커밋 버전(`8f5468e`)을 유지하여 실험의 재현성을 보장함)
  - `seed/` : 퍼징의 초기 입력값(Seed Corpus)이 저장된 공간
  - `harness/` : 타겟 프로그램과 AFL 퍼저를 연결해 주는 어댑터 코드가 위치함
  -  `*.sh` 스크립트 : 타겟 다운로드, AFL 빌드, 커버리지 측정 등을 자동화하는 실행 파일
- `Dockerfile`** : `docker-setup/` 내부의 스크립트들을 정해진 순서대로 실행하여, 독립적이고 동일한 환경을 자동으로 구축해줌
- `.gitmodules`** : AFL 등 외부 저장소를 현재 프로젝트의 서브모듈로 연결하기 위한 경로와 URL 정보가 기록된 파일

### 2. 실행 흐름
(1) 실험 환경 구축 
- 퍼징에 필요한 필수 패키지와 컴파일러를 설치함
- 이후 `setup_afl.sh`를 실행하여 서브모듈로 가져온 AFL 퍼저 자체를 빌드하여 사용 준비를 마침  
(2) 타겟 프로그램 준비 및 계측
- `download_benchmark.sh`를 통해 퍼징 타겟(Benchmark)의 소스코드를 다운로드
-  `build_with_afl.sh`를 통해 타겟을 **AFL 전용 컴파일러로 빌드(Instrumentation)** 함 (이를 통해 coverage 추적 가능)  
(3) 초기 입력값 및 테스트 환경 세팅
- 첫 입력값은 정상 입력(`seed/`)으로 세팅하고, 퍼저의 입력이 타겟 프로그램의 특정 함수로 잘 전달되도록 `harness/` 코드를 구성  
(4) 퍼징 실험
- `launch.sh` (또는 fuzzer-script 내부 실행 파일)를 가동함
- AFL이 시드 값을 기반으로 변이를 주며 타겟 프로그램에 입력을 넣고, 프로그램이 크래시되는 취약점을 찾아 `/output` 경로에 결과를 저장

### 추가 내용
- Git Submodule
  - Git 메인 저장소(Repository) 안에 **또 다른 독립적인 Git 저장소를 디렉터리 형태로 포함**시키는 기능
  - 단순히 소스 코드를 복사해서 붙여넣는 것이 아니라, 외부 저장소의 **특정 시점(Commit Hash)**을 가리킬 수 있음
  - 서브 모듈은 기본적으로 '하이퍼링크'이기 때문에, 프로젝트를 처음 가져올 때 소스코드가 자동으로 따라오지 않음
  - **Clone 직후:** 메인 프로젝트를 일반적인 `git clone`으로 가져오면, 서브모듈 폴더는 이름만 있고 속은 텅 빈 **빈 폴더** 상태가 됨
  - **Update 필수:** 이 상태로는 필요한 헤더 파일, 소스 파일 등이 없기 때문에 빌드(Build) 시 에러가 발생함. 빌드 전 반드시 명령어를 통해 연결된 실제 파일들을 다운로드(Sync)하여 속을 채워야 함
  - 필수 명령어
    - 처음 프로젝트 받을 때 한 번에 가져오기 : `git clone --recursive [메인 프로젝트 URL]`
    - 이미 clone 했는데 서브 모듈 폴더가 비어있을 때 : `git submodule update --init --recursive`
